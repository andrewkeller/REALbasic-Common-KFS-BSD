Unit Testing (Framework)

The Unit Testing Framework is a collection of classes that allow you to easily write and execute many, small, modular pieces of code that test something.

This page describes how to use the Unit Testing Framework, not why you should use test driven development, although some of the advantages of test driven development are scattered throughout this document.

Classes

UnitTestArbiterKFS

Class UnitTestArbiterKFS

Runs your test cases, and assembles results into a simple String.

The Unit Testing Framework is implemented using Introspection. You give a UnitTestArbiterKFS object a list of test classes, and it will identify and invoke the test cases automatically.

UnitTestBaseClassKFS

Class UnitTestBaseClassKFS

Provides a set of methods that greatly increase the ease of writing test cases. A set of test cases is contained within a class, and that class must be a subclass of UnitTestBaseClassKFS.

UnitTestExceptionKFS

Class UnitTestExceptionKFS

Raised when an assertion fails or when you don't catch an exception in a test case.

UnitTestWindowKFS

Class UnitTestWindowKFS

A Window that contains a UnitTestArbiterKFS object, a Listbox, and a Label, and is configured to display the results of unit tests interactively. It's plug-and-play, but it currently cannot display the results of alternate UnitTestArbiterKFS objects.

Compatibility

This framework is compatible with console applications, and should work on all platforms.

This framework is thread safe. The UnitTestBaseClassKFS class is inherently single threaded, but appropriate measures were taken to ensure proper operation in a multithreaded environment.

This framework is relatively decoupled from the rest of this library. It only requires four other classes: AutoreleaseStubKFS, DelegateClosureKFS, DurationKFS, and BSDGlobalsKFS_String.

Notes

In general, here's what happens:

  • Create a subclass of the UnitTestBaseClassKFS class.
  • Create your test cases in that class.
  • Submit an instance of your test class to a UnitTestArbiterKFS object.
  • If your code fails, then an exception of type UnitTestExceptionKFS is raised, and is automatically caught by the arbiter.
  • The arbiter keeps track of the failures, or lack thereof.
  • The arbiter raises the TestResultUpdated event to report new results.
  • In the UnitTestViewKFS container, the arbiter has already been configured to output the results to a listbox, so it makes for a nice little plug-and-play viewer for this framework.

To get your own unit tests running, here's what you need to do:

  • Put a UnitTestViewKFS on a window.
  • Create a subclass of the UnitTestBaseClassKFS class, and create your test cases in it.
    • See the documentation for the UnitTestBaseClassKFS class to see how test cases are written.
  • Instruct the arbiter in the UnitTestViewKFS container to run your tests (Assuming EnableAutomaticProcessing is set to True, then simply creating the jobs will cause them to get processed).

    myUnitTestViewKFS.Arbiter.CreateJobsForTestClasses( New TestMyGreatClass )
